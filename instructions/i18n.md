# Internationalization (i18n)

Standards for building software that works across languages and locales.

## Translatable Strings

- **[RULE]** Never concatenate strings to form translatable sentences. Use placeholders/interpolation so translators see the full sentence.

  Bad: `t('hello') + ' ' + name`
  Good: `t('hello_name', { name })`

- **[RULE]** Always use the project's translation functions (e.g., `t()`, `Trans`, `formatMessage()`). Never leave user-facing strings as bare literals.
- **[STRONG]** Provide translator context when a word is ambiguous (e.g., "post" as noun vs. verb). Use the i18n framework's context mechanism.
- **[STRONG]** Use proper pluralization functions (ICU plural syntax, or the framework's plural API). Do not use ternary operators to switch between singular and plural.

## RTL Support

- **[RULE]** Use logical CSS properties (`margin-inline-start`, `padding-inline-end`, `inset-inline`) instead of physical ones (`margin-left`, `padding-right`, `left`). This ensures layouts mirror correctly in RTL languages.
- **[STRONG]** Test layouts in RTL mode. Verify that icons, arrows, and directional elements flip appropriately.
- **[PREFER]** Use `dir="auto"` for user-generated content whose directionality is unknown.

## Locale-Aware Formatting

- **[STRONG]** Use `Intl.NumberFormat`, `Intl.DateTimeFormat`, and `Intl.RelativeTimeFormat` (or the project's localization utilities) for formatting numbers, dates, and times. Never hardcode date formats or thousand separators.
- **[PREFER]** Avoid assumptions about name order, address structure, or date conventions.

## Testing

- **[PREFER]** Test with long strings (German, Finnish) to verify layouts do not break with text expansion.
- **[PREFER]** Test with RTL languages (Arabic, Hebrew) to verify layout mirroring.
- Verify that translated strings render correctly in the UI -- not just that translation keys exist.
